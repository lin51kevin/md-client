## 1. 当前 MarkLite 已有功能总结

| 类别 | 具体功能 |

## 3. 差距分析 — 按性价比排序

### 🔴 第一梯队：高价值 · 低成本（强烈建议实现）

#### 1️⃣ 图片粘贴/拖拽插入
- **重要性**：⭐⭐⭐⭐⭐ 满分
- **实现复杂度**：中等
- **理由**：
  - 用户写文档最常见痛点
  - 支持从剪贴板粘贴图片自动保存到本地并插入 `![](path)` 链接
  - Typora / Obsidian / Mark Text 均已支持，这是用户切换工具的**头号原因**
  - Tauri 的 fs 插件可直接写文件，前端监听 paste 事件即可
  - **预计工作量**：5-8 天

#### 2️⃣ Toolbar 格式化按钮
- **重要性**：⭐⭐⭐⭐ 四星
- **实现复杂度**：低
- **理由**：
  - 加粗/斜体/标题/引用/链接/代码/列表等快捷格式化按钮
  - 当前 Toolbar 只有文件/视图操作，缺少编辑辅助
  - 对新手极其友好，实现简单（CodeMirror dispatch 插入标记即可）
  - **预计工作量**：2-3 天

#### 3️⃣ 文件树侧边栏
- **重要性**：⭐⭐⭐⭐ 四星
- **实现复杂度**：中等
- **理由**：
  - 打开一个目录后，左侧显示文件树，可导航和管理文件
  - 这是 Obsidian / Typora / Zettlr 的标配功能
  - 从「单文件编辑器」升级为「轻量级写作工具」的关键分水岭
  - Tauri 的 fs API 支持目录遍历
  - **预计工作量**：8-10 天

#### 4️⃣ 脚注支持
- **重要性**：⭐⭐⭐⭐ 四星
- **实现复杂度**：低
- **理由**：
  - remark 生态已有 `remark-footnotes` 插件
  - 只需添加到 remark 插件链即可
  - 学术和技术写作的必备功能
  - **预计工作量**：1-2 天

#### 5️⃣ YAML frontmatter 支持
- **重要性**：⭐⭐⭐⭐ 四星
- **实现复杂度**：低
- **理由**：
  - 添加 `remark-frontmatter` 插件
  - 编辑器中正确高亮 frontmatter 块
  - 预览中隐藏/渲染元数据
  - Hugo / Jekyll / Astro 等静态站点生成器用户的刚需
  - **预计工作量**：1-2 天

#### 6️⃣ 右键上下文菜单
- **重要性**：⭐⭐⭐ 三星
- **实现复杂度**：低
- **理由**：
  - 编辑器内右键菜单：复制/剪切/粘贴/全选/格式化操作
  - 桌面应用的基本体验
  - Tauri 支持原生菜单
  - **预计工作量**：2-3 天

### 🟡 第二梯队：高价值 · 中等成本（建议规划）

#### 7️⃣ 表格可视化编辑
- **重要性**：⭐⭐⭐⭐ 四星
- **实现复杂度**：中高
- **理由**：
  - Markdown 表格手写痛苦
  - 可提供插入表格对话框 + Tab 键跳转单元格 + 自动对齐
  - Typora 的杀手级功能之一
  - **预计工作量**：5-8 天

#### 8️⃣ 大纲支持 H4-H6
- **重要性**：⭐⭐⭐ 三星
- **实现复杂度**：低
- **理由**：
  - 当前只支持 H1-H3，长文档需要更细粒度导航
  - 改动极小，只需修改 toc.ts 的提取条件
  - **预计工作量**：1 天

#### 9️⃣ 自定义 CSS / 更多主题
- **重要性**：⭐⭐⭐ 三星
- **实现复杂度**：中
- **理由**：
  - 允许用户自定义预览样式或提供 3-5 套内置主题
  - 如 Notion 风格、学术风格、代码风格等
  - 提升差异化竞争力
  - **预计工作量**：5-7 天

#### 🔟 快捷键自定义
- **重要性**：⭐⭐⭐ 三星
- **实现复杂度**：中
- **理由**：
  - 允许用户在设置中重映射快捷键
  - 提升高级用户体验
  - **预计工作量**：4-6 天

#### 1️⃣1️⃣ 拼写检查
- **重要性**：⭐⭐⭐ 三星
- **实现复杂度**：中
- **理由**：
  - 至少支持英文拼写检查（可用浏览器原生 `spellcheck`）
  - 中文可选（可集成 `cjk-breaks` 等库）
  - 写作工具的标配
  - **预计工作量**：3-5 天

#### 1️⃣2️⃣ 跨文件搜索
- **重要性**：⭐⭐⭐ 三星
- **实现复杂度**：中
- **理由**：
  - 在文件树的基础上，支持在整个目录中搜索内容
  - Obsidian / VS Code 的核心功能
  - **预计工作量**：6-8 天

#### 1️⃣3️⃣ 导出为图片（PNG）
- **重要性**：⭐⭐⭐ 三星
- **实现复杂度**：中
- **理由**：
  - 将 Markdown 渲染结果导出为图片
  - 用于社交媒体分享
  - 可用 `html2canvas` 或 `puppeteer` 实现
  - **预计工作量**：4-6 天

#### 1️⃣4️⃣ 最近打开文件列表
- **重要性**：⭐⭐⭐ 三星
- **实现复杂度**：低
- **理由**：
  - 代码中已有 `recent-files.ts`，但用户手册未提及
  - 如已实现，需暴露到 UI（启动页/菜单）
  - **预计工作量**：1-2 天

### 🟢 第三梯队：中等价值 · 高成本（长期考虑）

#### 1️⃣5️⃣ 双向链接 `[[]]`
- **重要性**：⭐⭐⭐ 三星
- **实现复杂度**：高
- **理由**：
  - Obsidian 的核心卖点
  - 需要文件索引、自动补全、反向链接面板
  - 如果定位为笔记工具则必须有，如果定位为轻量编辑器则可跳过
  - **预计工作量**：15-20 天

#### 1️⃣6️⃣ 插件/扩展系统
- **重要性**：⭐⭐ 两星
- **实现复杂度**：极高
- **理由**：
  - Obsidian 生态的核心壁垒
  - 实现成本极高，但能让社区贡献功能
  - 初期不建议投入
  - **预计工作量**：30+ 天

#### 1️⃣7️⃣ WYSIWYG 所见即所得
- **重要性**：⭐⭐ 两星
- **实现复杂度**：极高
- **理由**：
  - Typora 的核心差异化
  - 需要替换或大改编辑器引擎（ProseMirror / TipTap）
  - 投入产出比低
  - MarkLite 保持分栏预览定位更合理
  - **预计工作量**：50+ 天

#### 1️⃣8️⃣ 实时协作
- **重要性**：⭐⭐ 两星
- **实现复杂度**：极高
- **理由**：
  - 需要后端服务 + CRDT/OT 算法
  - 与「轻量」定位冲突
  - **预计工作量**：60+ 天

#### 1️⃣9️⃣ 云同步
- **重要性**：⭐⭐ 两星
- **实现复杂度**：高
- **理由**：
  - 需要后端基础设施
  - 可引导用户使用 iCloud / Dropbox 等第三方同步
  - **预计工作量**：15-20 天

#### 2️⃣0️⃣ 标签/分类系统
- **重要性**：⭐⭐ 两星
- **实现复杂度**：中
- **理由**：
  - 依赖文件树和 frontmatter 基础设施
  - 在这两项完成后再考虑
  - **预计工作量**：8-10 天

## 5. 核心结论

### ✨ MarkLite 的差异化优势

1. **Tauri 原生桌面应用** — 性能好，资源占用少
2. **三种焦点模式** — 打字机 + 专注 + 全屏，竞品中较少见
3. **三格式导出** — DOCX / PDF / HTML，竞品多数只支持 PDF
4. **丰富的预览引擎** — Mermaid + KaTeX + 自定义容器，开箱即用
5. **分栏同步滚动** — 实时预览体验流畅

**MarkLite 在开源轻量编辑器中已是第一梯队。**

---

### 🚨 最大短板

**缺少图片处理和文件树** — 这两个功能是用户从 Typora / Obsidian 迁移过来的最大阻碍。

### 💡 为什么图片粘贴是最高优先级

用户写文档工作流：
1. 打开编辑器（MarkLite ✅）
2. 编写内容（MarkLite ✅）
3. 插入图片 → **MarkLite ❌ Typora ✅** ← **用户放弃 MarkLite**
4. 导出文档（MarkLite ✅）

图片粘贴是放弃率最高的卡点，也是实现最简单的功能。**投入产出比最高。**

---

### 🎯 定位建议

**不要追 Obsidian 的知识图谱路线**（成本太高，生态竞争激烈）

**走 Typora 开源替代品路线**：
- ✅ 专注单文件/小目录的轻量 Markdown 写作体验
- ✅ 强调导出能力（DOCX/PDF/HTML）
- ✅ 强调沉浸式写作（焦点模式）
- ✅ 强调编辑效率（自动保存、快捷键、格式化按钮）
- ❌ 不追求笔记关联和知识图谱
- ❌ 不追求云同步和实时协作
